### 1、top -c 命令查看操作系统整理情况

![](.\xingneng\top.jpg)

关注几个核心参数：

第一行：系统运行时间 & 负载

核心参数load average：近1/5/15 分钟操作系统平均负载，当负载数 <≈ CPU 核数（小于或约等于）时是正常的。

第二行：进程状态（Tasks）

第三行：CPU 使用率（核心）


| 字段 | 含义         | 常见问题             |
| ------ | -------------- | ---------------------- |
| us   | 用户态 CPU   | 程序计算多           |
| sy   | 内核态 CPU   | 系统调用 / IO        |
| ni   | nice         | 可忽略               |
| id   | 空闲 CPU     | 越高越好             |
| wa   | **IO 等待**  | **磁盘慢 / 网络 IO** |
| hi   | 硬中断       | 网卡/硬件            |
| si   | 软中断       | 网络包多             |
| st   | 被偷走的 CPU | 虚拟机资源争抢       |

**快速判断法**

- **us 高** → 应用算力瓶颈
- **sy 高** → 内核 / IO 问题
- **wa 高（>10%）** → **磁盘 IO 瓶颈**
- **id 很低** → CPU 不够

第四行：内存使用

**内存不是越 free 越好**


| 字段       | 含义           |
| ------------ | ---------------- |
| free       | 真正空闲       |
| used       | 已用           |
| buff/cache | 缓存（可回收） |

第五行：Swap

进程列表字段详解

```
PID USER  PR  NI  VIRT  RES  SHR S  %CPU %MEM TIME+  COMMAND
```


| 字段    | 含义             | 关注点     |
| --------- | ------------------ | ------------ |
| PID     | 进程 ID          | 定位用     |
| USER    | 用户             |            |
| PR      | 优先级           |            |
| NI      | nice             |            |
| VIRT    | 虚拟内存         | 包含 swap  |
| RES     | **实际物理内存** | **最重要** |
| SHR     | 共享内存         |            |
| S       | 状态             | R / S / D  |
| %CPU    | CPU 占比         |            |
| %MEM    | 内存占比         |            |
| TIME+   | CPU 累计时间     |            |
| COMMAND | 命令             |            |

进程状态 


| 状态 | 含义                         |
| ------ | ------------------------------ |
| R    | 运行中                       |
| S    | 休眠                         |
| D    | **不可中断 IO 等待（危险）** |
| Z    | 僵尸                         |



### 2、排查典型场景速查

#### 2.1 系统慢，但 CPU 不高

四看：

（1）看 `%wa`，看topc 命令输出第三行，CPU使用率，IO 等待指标是否过大（>10%），若存在，则为IO瓶颈

（2）看是否 `D` 状态进程，看topc 命令输出进程列表，是否存在`D`状态进程，若存在，存在IO等待进程

（3）看操作系统IO情况，使用iostat命令详细查看操作系统IO负载情况，深入定位

```
iostat -x 1 3
```

含义：

- `-x`：显示**扩展指标（必须）**
- `1`：每 1 秒刷新一次
- `3`：打印三次

输出样例：

![](.\xingneng\iostat.jpg)

```
Linux 5.10 (server01)  01/21/2026  _x86_64_

avg-cpu:  %user  %nice %system %iowait  %steal %idle
           6.12    0.00    2.31    18.45    0.00   73.12

Device            r/s   w/s   rkB/s   wkB/s  rrqm/s wrqm/s await  r_await  w_await  svctm  %util
sda              120.3  85.6   5200    3400     2.1   1.3  45.6     38.2     56.4     3.2   98.7
```

**avg-cpu关键字段**

| 字段        | 含义        | 判断                           |
| ----------- | ----------- | ------------------------------ |
| %user       | 用户态 CPU  | 应用计算                       |
| %system     | 内核态 CPU  | IO/系统调用                    |
| **%iowait** | **IO 等待** | **磁盘慢的直接信号，核心指标** |
| %idle       | 空闲        |                                |

**核心指标%iowait > 10% 就要警惕**

**Device 关键字段**

| 字段              | 含义                                                | 判断                                                         |
| ----------------- | --------------------------------------------------- | ------------------------------------------------------------ |
| %util             | 磁盘繁忙程度，磁盘是否被打满（第一指标）            | < 50% 很空闲，50%～80%有压力，\> 80%明显瓶颈，≈100%**磁盘打满（必慢）** |
| await             | IO 总等待时间，一次 IO 从发出到完成的平均时间（ms） | 正常范围SSD硬盘 < 5～10 ms，HDD硬盘<< 20～50 ms， **await 高 = IO 慢** |
| r_await / w_await | 读写等待时间                                        | 判断是 **读慢** 还是 **写慢**，数据库 / 日志场景非常有用     |
| r/s、w/s          | IO 次数，每秒读 / 写请求数                          | r/s、w/s 高 + await 高 = **IO 请求堆积**                     |

典型场景：

**场景 1：%util 100%，await 高**

 磁盘已满载，所有进程排队

**场景 2：%util 不高，但 await 高**

可能是：

- 单次 IO 很大
- RAID / 网络存储
- 磁盘抖动

**场景 3：%iowait 高，CPU 很闲**

CPU 在等磁盘

**场景 4：写 await 明显高于读**

常见于：

- 日志刷盘
- 数据库 checkpoint
- fsync 频繁

（4）使用iotop命令，找“罪魁祸首进程”

```
iotop -o -d 1
```

参数：

 -o 只看正在IO的进程

-d 每秒刷新一次

命令输出样例：

![](.\xingneng\iotop.jpg)

顶部两行：系统整体 IO

DISK READ / WRITE 进程每秒读写多少数据

IO% IO等待比例，表示该进程有多少时间在等 IO

| IO%    | 说明        |
| ------ | ----------- |
| < 10%  | 正常        |
| 10~30% | 有等待      |
| > 50%  | IO 阻塞明显 |
| > 80%  | 严重阻塞    |
